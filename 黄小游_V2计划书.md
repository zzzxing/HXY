# 黄小游 V2 版本计划书

## 一、V2 的定位调整

当前版本已经完成了一个能跑通主链路的 MVP：教师可创建活动、配置点位与任务，学生可提问、上传证据，系统可生成学习档案，教师可进行查看与评价。这说明底层业务主干已经具备雏形。

但从产品形态上看，当前版本更像“教学管理原型”，还不像“适合信息技术课程上使用的数字研学平台”。V2 的核心目标不是继续堆后台功能，而是把产品升级为：

**面向信息技术课堂、支持“足不出教室也能开展数字研学”的沉浸式学习平台。**

V2 的产品关键词：

- 数字研学
- 虚拟导览
- 课堂可用
- 场景探索
- 任务驱动
- 证据沉淀
- AI 引导
- 小组协作
- 展示与评价

---

## 二、对当前版本的诊断

### 1. 已有优点

当前版本已经具备以下可继续利用的基础：

- 有清晰的角色体系：管理员 / 教师 / 学生
- 有基本的活动、点位、任务、证据、档案数据流
- 有 AI 提问入口
- 有图片与文字证据上传能力
- 有教师端查看与评价机制
- 项目已能以 Python FastAPI 方式运行，适合继续迭代

### 2. 主要问题

#### （1）UI 过于后台化

当前页面整体偏 Bootstrap 管理系统风格，信息是“列出来了”，但缺乏视觉重点、场景感和探索感。学生进入后更像在用表单系统，而不是在进入一个数字研学空间。

#### （2）学生体验不够“研学化”

当前“游”板块更接近普通详情页：点位介绍、问题链文本、证据上传表单。它没有把“探索—发现—记录—解释”的过程做成明确交互路径。

#### （3）课堂场景适配度不够

你这个产品的真实使用场景并不一定是外出实地研学，也包括信息技术课堂中的“虚拟研学”“数字博物馆参访”“线上点位探索”。V1 对这一点体现不足。

#### （4）功能虽然有链路，但不够有记忆点

当前功能能跑，但“亮点功能”不足，缺少让评审、老师、学生一眼记住的产品特征。

#### （5）教师端更像配置后台，缺少教学驾驶舱

教师端现在偏“增删改查”，但课堂中教师更关心的是：

- 学生是否进入了探索状态
- 哪些点位最难
- 哪些问题最有价值
- 哪些学生没跟上
- 哪些证据值得课堂展示

V1 在这方面不够强。

#### （6）仓库结构还不够收敛

当前仓库中同时存在 Python/FastAPI 路线和前端工程骨架，V2 应统一为一条主技术路线，避免项目看起来像“半成品混搭工程”。

---

## 三、V2 的目标

V2 目标不是“做得更多”，而是“做得更像一个产品”。

### 总目标

将黄小游升级为一个适用于信息技术课程、支持虚拟导览与任务探究、兼具审美与课堂组织能力的数字研学平台。

### 分目标

1. 让首页一眼看出“这是数字研学平台”，而不是后台系统
2. 让学生端从“填表上传”变成“探索式学习”
3. 让教师端从“配置页”升级为“课堂驾驶舱”
4. 让 AI 从单一问答入口升级为“导游 + 探究教练”双角色
5. 让课堂内使用也成立，不依赖必须外出
6. 让评审一眼能看到亮点，不再只是基础 CRUD

---

## 四、V2 设计原则

### 1. 场景优先，而不是表单优先

学生首先看到的应该是“目的地、场景、路线、任务、热点”，而不是一排排功能按钮。

### 2. 探索优先，而不是阅读优先

页面不能只是文字说明，应尽量通过大图、分层卡片、热点按钮、任务节点、时间线、证据墙来组织内容。

### 3. 课堂优先，而不是旅游优先

这是“教育型数字研学平台”，不是旅游 App。所有设计都应服务于课堂任务、问题链和证据链。

### 4. 学习过程可见

学生不是只提交一个结果，而要让“问题—观察—证据—解释—成果”全过程可视化。

### 5. 教师可控

教师必须能配置、看到、筛选、干预，不能把系统做成不可控的纯 AI 产品。

### 6. 轻量可演示

V2 仍应优先保证“开发可落地、演示效果强、主流程跑得通”，不要一口气做太重的 AR/VR 大系统。

## 5.2 页面视觉重构

### UI 风格建议

主基调：
- 清爽、现代、教育感、探索感
- 不是旅游海报风，也不是后台管理风

建议视觉关键词：
- 蓝绿主色
- 橙色强调
- 浅底卡片
- 圆角大卡
- 插画化图标
- 地图化布局
- 进度可视化

### 重点页面重构

#### （1）登录页
从普通登录框改为更有产品气质的欢迎页：
- 左侧品牌语
- 右侧登录卡
- 背景可加入城市/博物馆/地图元素

#### （2）学生首页
从列表页改为“数字研学大厅”：
- 本期推荐路线
- 热门展馆/城市/专题
- 我的班级任务
- 快捷继续上次学习

#### （3）点位页
点位页必须从“文字详情页”升级为“探索页”：
- 顶部大图 / 360 场景入口
- 热点按钮
- AI 导游入口
- 探究提示卡
- 证据上传浮层
- 完成度进度条

#### （4）学习档案页
从普通列表升级为“时间线 + 成就墙 + 证据墙”：
- 我的问题演进
- 我的证据合集
- 我的结论卡
- 教师评语
- 评分与徽章

#### （5）教师端首页
升级为教学驾驶舱：
- 班级参与率
- 点位完成热力
- 高频问题词云
- 优秀证据推荐
- 待点评学生
- 当前活动概览

## 六、V2 功能扩展方案

下面这些功能不是平均铺开做，而是按“高价值 + 强演示 + 适合课堂”优先级筛选出来的。

### A 级：必须做

#### 1. 虚拟地图 / 数字展馆入口
让学生不出门也能“进入场景”。

可先做轻量版：
- 目的地地图页
- 点位节点卡
- 路线解锁
- 大图热点跳转

#### 2. 点位热点探索
每个点位不只是文字介绍，而是：
- 热点 1：看重点
- 热点 2：听讲解
- 热点 3：提问题
- 热点 4：收集证据

#### 3. AI 双模式
目前 AI 更像通用问答入口。V2 建议分为：
- AI 导游：讲解场景、回答“这是什么”
- AI 探究教练：引导观察、比较、提出假设、提醒记录证据

#### 4. 证据背包系统
把“上传证据”做成学生的收藏背包，而不是单纯表单提交。

背包内包含：
- 图片证据
- 文字证据
- 问题卡
- 猜想卡
- 结论卡
- 待验证线索

#### 5. 身份任务 / Quest 机制
参考有效的 field trip 任务设计，把学生变成有身份的探究者：
- 小记者
- 城市观察员
- 工业遗产解码员
- 博物馆策展人
- 历史证据搜集员

这样做能极大增强课堂参与感。

#### 6. 成就与徽章
加入低成本但强反馈的激励机制：
- 首次提问
- 首次证据上传
- 证据达人
- 追问之星
- 最佳观察者
- 小组协作者

### A- 级：优先做

#### 7. 小组协作页
学生可以以小组为单位完成：
- 分工卡
- 小组共用证据池
- 小组成果海报

#### 8. 教师精选展示
教师可把优秀问题、优秀证据、优秀成果置顶到“课堂展示区”。

#### 9. 过程性学情分析
教师端提供：
- 参与率
- 点位完成率
- 高频疑问点
- 证据提交统计
- 待干预学生提醒

### B 级：可预留接口

#### 10. OCR / 语音转写
不一定 V2 就做深，但建议预留接口：
- 图片转文字
- 语音转文本
- 自动提取关键词

#### 11. 轻量 360 / 全景
V2 可先支持“全景图入口”，不一定要自己做复杂全景编辑器。

#### 12. 课堂大屏模式
教师一键投屏展示：
- 全班热词
- 点位完成状态
- 优秀作品轮播

## 八、V2 技术路线建议

## 8.1 主技术路线建议

V2 建议**统一为 Python FastAPI + Jinja2/前端模板增强**，先把一条路线走通，不要继续维持混合技术栈的摇摆状态。

### 推荐原因

- 你当前已经有可运行的 Python 版本
- 你是零基础，更适合先在一条技术链上稳定推进
- V2 的重点在产品体验和功能重构，不在于技术炫技

### 前端增强建议

在不完全推翻 Python 架构的前提下，可引入：
- 更完整的自定义 CSS 设计系统
- 少量 Alpine.js / HTMX / 原生 JS 增强交互
- Chart.js 做教师端可视化
- Lightbox / Carousel 做点位大图展示

这样可以显著提升体验，而不必立刻切成重型前后端分离项目。

## 8.2 仓库结构建议

V2 应精简主目录，尽量收敛到：

```text
README.md
requirements.txt
.env.example
pyapp/
  main.py
  config.py
  database.py
  models.py
  services/
  routes/
  templates/
  static/
  scripts/
uploads/
```

可保留 `legacy/` 目录暂存旧代码，但不要让主目录同时出现两套并行产品骨架。

## 十、V2 成功标准

V2 完成后，应达到以下状态：

### 产品感
- 学生一进入就知道这是“数字研学平台”
- 页面审美明显优于 V1
- 交互有探索感，而不是填表感

### 教学感
- 支持课堂里“足不出教室也能研学”
- 支持任务驱动、问题链、证据链
- 教师能看到过程、组织课堂、展示成果

### 演示感
- 有明显亮点页
- 有可讲述的功能故事
- 能打动评审和老师

### 工程感
- 目录更清晰
- 启动更简洁
- 不再是混合式半成品结构

---

## 十一、建议你对 Codex 的开发策略

不要再让它“自由发挥做大而全系统”，而是要强约束：

1. 先重构产品结构和 UI
2. 再补核心亮点功能
3. 严禁只会做后台列表和表单
4. 严禁只改颜色不改交互
5. 严禁为了快而继续堆技术栈
6. 严禁写死假数据冒充沉浸式功能

V2 的关键不在于“多”，而在于：

**更像产品、更像课堂工具、更像数字研学平台。**
