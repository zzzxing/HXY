# 黄小游研学智能体开发总说明

## 1. 这个软件要做什么

“黄小游”不是一个普通聊天软件，而是一个围绕**研学前—研学中—研学后**全过程运行的教育软件。  
它要把课题文档里提出的“学—研—游”流程，真正变成一个可用的软件系统。

软件的核心目标不是“炫技”，而是解决下面几个真实问题：

1. 学生出发前不知道要学什么、问什么。
2. 家庭探究容易变成随便搜一搜、抄一抄。
3. 现场参观容易走马观花，问题跟不上、证据留不住。
4. 研学结束后缺少学习档案，教师难评价，学生难复盘。

所以，软件的核心价值是：

- 帮教师组织研学任务
- 帮学生形成问题链
- 帮学生在现场记录证据
- 帮系统把“问题—任务—证据—成果”串起来
- 最后生成学习档案，便于评价和展示

---

## 2. 建议你先做什么版本

你是编程零基础，而且准备交给 Codex 来做，所以**第一版绝对不要做太重**。

建议第一版做成：

- **手机优先的 Web App（H5/PWA）**
- 角色包括：
  - 学生端
  - 教师端
  - 管理后台
- 暂时不单独做：
  - 家长独立端
  - 讲解员独立端
  - 复杂审批流
  - 复杂数据大屏
  - 真正的多模型编排平台

这样做的好处：

1. 技术难度明显更低。
2. Codex 更容易一次性生成能跑通的版本。
3. 后面可以继续扩展成小程序、App 或多端系统。
4. 比赛/课题展示时也足够完整。

---

## 3. 第一版必须包含的能力

### 3.1 教师端
- 创建一次研学活动
- 设置活动主题、对象、时间、路线
- 配置点位、任务、问题链
- 查看学生提交的证据与成果
- 导出学习档案

### 3.2 学生端
- 查看自己的研学活动
- 进行课前导学问答
- 生成和保存个人问题清单
- 查看家庭探究任务
- 到现场扫码进入点位学习卡
- 上传照片/文本/录音作为证据
- 与 AI 进行点位相关问答
- 查看个人学习档案

### 3.3 管理后台
- 管理学校、班级、教师、学生
- 管理研学主题、点位、知识库
- 生成点位二维码
- 配置 AI 提示词模板
- 管理敏感词、日志与内容审核规则

---

## 4. 第一版先不做的能力

这些内容很重要，但不适合第一版上来就做：

- 实时语音对讲式 AI
- 自动识别所有图片内容并精准抽取证据
- 家长协同独立账号体系
- 讲解员现场联动工作台
- 复杂推荐算法
- 大规模并发
- 多学校 SaaS 商业化结算
- 离线 AI 模型本地部署
- AR / VR / 元宇宙功能

这些可以写进“后续版本规划”，但不要作为第一版必做。

---

## 5. 我建议的技术路线

为了让零基础也更容易推进，建议技术栈尽量简单统一：

### 前端
- Next.js
- TypeScript
- Tailwind CSS
- shadcn/ui

### 后端
- Next.js API Routes / Server Actions
- Supabase（数据库 + 登录认证 + 文件存储）

### AI 能力
- OpenAI 兼容接口
- RAG 简化版：先做“点位知识 + 活动上下文”注入，不一开始做复杂向量检索
- 后续再加向量检索

### 文件能力
- 图片上传
- 录音上传
- 文本输入
- 后续可扩展 OCR / ASR

### 部署
- Vercel 部署前端和接口
- Supabase 托管数据库和存储

---

## 6. 你给 Codex 的工作方式

不要只丢一句“帮我做一个研学智能体”。  
正确做法是把这套文档分阶段交给 Codex：

1. 先让它搭项目骨架
2. 再做数据库和登录
3. 再做教师端
4. 再做学生端
5. 再接 AI
6. 再做后台
7. 最后做测试和修复

这样成功率会高很多。

# 黄小游研学智能体 PRD（产品需求文档）

## 1. 产品名称

黄小游 —— AI 研学智能体

---

## 2. 产品定位

面向中小学研学旅行场景的 AI 辅助学习系统。  
围绕“学—研—游”全过程，为教师提供活动组织与评价工具，为学生提供问题引导、任务执行、现场学习和证据沉淀工具。

---

## 3. 产品目标

### 3.1 业务目标
1. 让研学从“参观+讲解”升级为“问题驱动+证据化学习”。
2. 为学校提供可复制的研学数字化支撑工具。
3. 降低教师组织研学的复杂度。
4. 提高学生课前准备、现场探究和课后复盘质量。

### 3.2 用户目标
#### 教师
- 方便创建和管理研学活动
- 快速配置点位任务和问题链
- 能看见学生过程数据和成果
- 能导出学习档案用于评价

#### 学生
- 知道出发前要学什么
- 知道到现场要看什么、问什么、记什么
- 能方便记录照片、文字、录音
- 能获得 AI 的简明讲解和追问引导
- 能看到自己的成长轨迹

- 中小学教师

### 4.2 次级用户

### 4.3 暂不作为首版重点
- 家长独立用户
- 基地讲解员独立用户

---

## 5. 使用场景

### 场景一：课前导学
教师发布研学活动，学生进入活动页面，浏览导学包，与 AI 进行问答，形成个人问题清单。

### 场景二：家庭探究
学生根据系统生成的任务单完成资料收集，填写资料卡，整理“待验证问题”。

### 场景三：现场点位学习
学生到达研学点位，扫描二维码进入点位学习卡，查看短讲解、问题链和证据清单，拍照/录音/记录证据，并继续向 AI 提问。

### 场景四：课后总结
系统自动汇总问题、任务、证据与成果，生成学习档案，教师按量规进行评价。

---

## 6. 核心产品理念

### 6.1 四链闭环

- 证据链

### 6.2 软件不是替学生答题

# 黄小游研学智能体技术方案与系统架构

## 1. 技术选型原则

本项目优先遵循以下原则：

1. **先跑通，再复杂化**
2. **少后端、少运维、少自建**
3. **适合 Codex 自动生成**
4. **适合零基础后续维护**
5. **支持手机优先访问**

---

## 2. 推荐技术栈

## 2.1 前端
- Next.js 14+
- TypeScript
- Tailwind CSS
- shadcn/ui
- React Hook Form
- Zod

## 2.2 后端
- Next.js Route Handlers / Server Actions
- Supabase Auth
- Supabase Postgres
- Supabase Storage

## 2.3 AI 接入
- OpenAI 兼容接口
- 模型通过环境变量配置：
  - `OPENAI_BASE_URL`
  - `OPENAI_API_KEY`
  - `OPENAI_MODEL`

## 2.4 工具能力
- 二维码生成：`qrcode`
- 二维码扫描：`html5-qrcode`
- 文件上传：Supabase Storage
- 日期处理：`dayjs`

## 2.5 部署
- Vercel：前端与 API
- Supabase：数据库与存储

---

## 3. 整体架构

```text
[学生端 / 教师端 / 后台]
          |
       Next.js
          |
  -----------------------
  |         |           |
Auth      API         AI Service
  |         |           |
Supabase  Postgres   OpenAI-compatible API
  |         |
Storage   Business Logic
```

## 5. AI 实现建议

## 5.1 第一版不要做太复杂的 RAG
第一版建议做“轻量上下文注入”：

- 活动标题
- 活动简介
- 点位名称
- 点位知识简介
- 当前任务
- 问题链模板
- 安全提示规则

把这些作为 prompt 上下文传给模型即可。

### 示例
```text
你是“黄小游”，一个面向中小学生的研学学习助手。
当前活动：黄石工业文化研学
当前点位：黄石矿博物馆展厅A
当前任务：观察矿石标本，记录两种差异
回答要求：
1. 不直接替学生完成最终结论
2. 多用引导式提问
3. 回答简短清楚
4. 优先提醒学生去观察、比较、记录证据
```

## 5.2 第二版再引入向量检索
后续可增加：
- pgvector
- 向量化点位知识
- 相似段落召回
- 引用来源片段

---

## 6. 文件与媒体处理策略

## 6.1 第一版支持
- 图片上传
- 录音上传
- 文本输入

## 6.2 第一版处理方式
- 图片先只做存储，不强依赖自动识别
- 录音先做上传，不强依赖自动转写
- 用户可以补充文字说明

## 6.3 第二版扩展

## 7. 安全与治理设计

## 7.1 未成年人保护
- 不展示无关外部链接
- 禁止自由生成敏感内容
- 聊天记录留痕
- 可限制问答长度和频率

## 7.2 数据安全
- Row Level Security
- 上传文件按用户路径隔离
- 重要操作审计日志

## 7.3 内容安全
- 敏感词过滤
- 系统提示词限制回答范围
- 不允许 AI 给出不适龄内容

# 黄小游研学智能体数据库设计

## 1. 设计原则

1. 先满足 MVP
2. 表结构保持清晰
3. 角色、活动、点位、任务、证据、档案主线明确
4. 便于后续扩展家长端和讲解员端

---

## 2. 核心实体关系

```text
学校 school
  └── 班级 class
        └── 学生 student_profile

教师 teacher_profile
  └── 活动 activity
        ├── 点位 activity_site
        ├── 任务 task
        ├── 学生参与 activity_member
        ├── 学生问题 question_item
        ├── 学生证据 evidence
        └── 学习档案 portfolio
```

---

## 3. 表清单

## 3.1 profiles
统一用户基础表，和 Supabase Auth 用户对应。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键，关联 auth user |
| role | text | admin / teacher / student |
| name | text | 姓名 |
| school_id | uuid | 学校ID |
| class_id | uuid | 班级ID，教师可为空 |
| student_no | text | 学号，可为空 |
| created_at | timestamptz | 创建时间 |

---

## 3.2 schools
| 字段 | 类型 | 说明 |

## 2. 项目目标

这个系统用于支持中小学研学活动的“学—研—游”全过程。  
不是普通聊天软件，而是一个**任务驱动 + 点位学习 + 证据沉淀 + 学习档案生成**的教育系统。

---

## 3. 必须实现的 MVP 功能

### 3.1 角色
- admin
- teacher
- student

### 3.2 学生端
- 登录
- 查看活动列表
- 查看活动详情
- 学板块：AI 导学问答 + 保存问题清单
- 研板块：查看任务 + 填写资料卡/待验证问题
- 游板块：扫码或模拟进入点位学习卡
- 点位页：查看简介、问题链、证据清单、AI 问答
- 上传证据（图片/文字/录音至少支持图片+文字）
- 查看学习档案

### 3.3 教师端
- 创建活动
- 编辑活动
- 配置点位
- 配置任务
- 查看学生过程数据
- 查看并评价学习档案

### 3.4 后台
- 用户管理基础页面
- 学校/班级基础页面
- 资源模板基础页面

## 4. 技术要求

### 4.1 技术栈
- Next.js 14+
- TypeScript
- Tailwind CSS
- shadcn/ui
- Supabase（Auth / Postgres / Storage）

### 4.2 代码要求
- 使用 App Router
- 使用模块化目录结构
- 类型定义清晰
- 表单校验使用 Zod
- 数据访问封装在独立 service 层
- 不要把所有逻辑堆在页面里

### 4.3 UI 要求
- 手机优先
- 现代简洁
- 教育产品风格
- 支持浅色主题即可

## 7. AI 功能要求

### 7.1 第一版 AI 方案
请使用 OpenAI 兼容接口，不要自建复杂智能体编排系统。  
先实现一个简单的服务函数：

- 输入：活动信息、点位信息、阶段、用户提问
- 输出：简短、适合学生的引导式回答

### 7.2 AI 回答限制
- 不直接替学生完成整份成果
- 多用启发式提问
- 答案简洁
- 优先提醒观察、比较、记录证据
